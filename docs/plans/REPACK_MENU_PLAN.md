# Repack 계획서 (메뉴 우선)

## 1. 목표
- 수집, 라벨링, 글 작성, 발행을 완전히 독립 실행 가능한 구조로 재편한다.
- 설정은 전역 통합 메뉴가 아니라 각 도메인 하위(2단계) 메뉴로 배치한다.
- 운영자가 "기능 선택 -> 실행/설정" 흐름으로 즉시 접근 가능하게 만든다.

## 2. 정보구조(IA)

### 2.1 1단계 메뉴(Primary)
1. 대시보드
2. 키워드 관리
3. 수집
4. 라벨링
5. 글 작성
6. 발행
7. 로그/모니터링

### 2.2 2단계 메뉴(Secondary)

#### 2.2.1 대시보드
1. 통합 현황
2. 단계별 상태

#### 2.2.2 키워드 관리
1. 카테고리/키워드
2. 연관 키워드
3. 차단 키워드

#### 2.2.3 수집
1. 수집 실행
2. 수집 설정
3. 작업 이력
4. 수집 데이터

#### 2.2.4 라벨링
1. 라벨링 실행
2. 라벨링 설정
3. 라벨링 결과

#### 2.2.5 글 작성
1. 글 작성 실행
2. 글 작성 설정
3. 페르소나 관리
4. 템플릿 관리
5. AI API 관리
6. 작성 결과/에디터

#### 2.2.6 발행
1. 발행 실행
2. 발행 설정
3. 발행 채널 관리
4. 발행 이력

#### 2.2.7 로그/모니터링
1. 실행 로그
2. 실패 로그
3. 재시도 큐

## 3. 실행 분리 원칙
1. 수집, 라벨링, 글 작성, 발행은 서로 직접 호출하지 않는다.
2. 단계 간 연결은 함수 호출이 아닌 DB 상태 기반으로만 처리한다.
3. 각 단계는 단독 실행/재실행이 가능해야 한다(idempotent).
4. 입력 조건 미충족 시 안전한 no-op + 명시적 메시지를 반환한다.

## 4. 설정 정책

### 4.1 수집 설정
- 채널 활성/비활성
- 키워드 범위(선택/카테고리/전체)
- max_results, timeout, retry
- 연관 키워드 동기화 옵션(네이버)

### 4.2 라벨링 설정
- 라벨링 방식(rule/ai)
- 배치 크기
- 품질 임계값
- 재라벨링 정책

### 4.3 글 작성 설정
- 기본 템플릿
- 기본 페르소나
- AI 제공자 우선순위
- 최소 소스 개수

### 4.4 발행 설정
- 채널별 발행 모드(수동/자동)
- 발행 주기
- 실패 재시도 정책
- 승인 필요 여부

## 5. 화면 동작 규칙
1. 1단계 메뉴 클릭 시 해당 도메인 기본 2단계 화면으로 진입한다.
2. 2단계 "설정" 화면의 저장은 해당 도메인 설정만 변경한다.
3. 각 "실행" 화면은 자기 도메인 로그만 보여준다.
4. 다른 단계 상태를 읽을 수는 있지만 직접 실행 트리거하지 않는다.

## 6. 이행 계획(코드 작업 전 설계 단계)

### Phase 0: 설계 확정
1. 메뉴/네비게이션 와이어프레임 확정
2. 단계별 상태 모델 확정
3. 설정 키 네임스페이스 확정 (collect.*, label.*, writer.*, publish.*)

### Phase 1: UI 구조 리팩토링
1. 상단/사이드 1단계 메뉴 재배치
2. 각 도메인 2단계 탭/서브메뉴 추가
3. 기존 통합 설정 화면 분해

### Phase 2: 실행/설정 분리
1. 단계별 실행 핸들러 분리
2. 단계별 설정 저장 API 분리
3. 단계별 로그 조회 API 분리

### Phase 3: 운영 안정화
1. 단독 실행 테스트
2. 역순 실행 테스트(발행 먼저 실행 등)
3. 반복 실행/중복 방지 테스트
4. 장애 복구 시나리오 검증

## 7. 수용 기준(Definition of Done)
1. 각 도메인에 실행+설정 화면이 분리되어 존재한다.
2. 어느 단계도 다른 단계를 내부 호출하지 않는다.
3. 단계별 로그/이력/실패 원인이 화면에서 분리 조회된다.
4. 운영자가 단계별로 독립 실행과 재실행을 수행할 수 있다.
